# DevScout — 채용 공고 자동 알림 봇 기획서 (Claude Code 전달용)

> 프로젝트명: **DevScout** (레포명: `devscout`)

## 1. 목표

조건에 맞는 신규 개발자 채용 공고(특히 신입 공채)를 여러 사이트에서 자동으로 수집해 Discord로 알림을 보낸다. 매일 여러 사이트를 돌아다니며 확인하던 시간을 없애는 것이 목적이며, 동시에 포트폴리오에 넣을 수 있는 사이드 프로젝트이기도 하다.

## 2. 대상 사이트

요청한 5곳(원티드, 점핏, 사람인, 잡코리아, 랠릿, 로켓펀치)에 아래 2곳을 추가로 추천한다.

- **캐치(catch.co.kr)** — 대기업 공채 일정 전문. "신입 공채"를 따로 챙기고 싶다면 여기가 핵심.
- **자소설닷컴(jasoseol.com)** — 대졸/신입 특화, 채용달력(공채 마감 카운트다운)이 강점. 캐치와 겹치는 대기업 공채 영역이지만 실시간 지원자 수·합격자 데이터랩 등 부가 정보가 차별점. 개발자 전용 공고보다는 대기업 정기공채 위주라 원티드·점핏·랠릿을 보완하는 용도로 추가.

(참고: 프로그래머스 채용은 2025.06.23부로 서비스가 종료돼 대상에서 제외했습니다.)

사이트별로 HTML 구조가 다 다르고, 공식 검색 API를 공개하지 않는 곳이 대부분이라 사이트마다 별도 파서(parser) 모듈이 필요하다. 처음부터 8곳을 다 하려 하지 말고 2곳(예: 원티드, 점핏)으로 끝까지 만들어본 뒤 나머지를 같은 패턴으로 추가하는 게 2~3일 안에 끝내는 현실적인 방법이다.

## 3. 아키텍처

```
[스케줄러: cron 또는 GitHub Actions schedule]
        ↓ (예: 매일 아침 8시, 저녁 8시)
[사이트별 크롤러 모듈] (site-adapter 패턴, 사이트당 1개 파일)
        ↓ 원본 공고 리스트
[필터링] — 키워드 매칭 (Node.js, 백엔드, 신입, 부트캠프 우대 등) + 신입공채 여부 태깅
        ↓
[중복 체크] — DB에 이미 저장된 공고 ID/URL이면 스킵 (Prisma + PostgreSQL/SQLite)
        ↓ 새 공고만
[Discord Webhook 알림 전송]
```

- **크롤링 도구**: 대부분 JS 렌더링 기반 SPA라 Cheerio+axios만으로는 부족할 가능성이 높음. Playwright(headless) 권장. Node.js 스택 그대로 재사용 가능.
- **DB**: 이력서에 이미 있는 Prisma + PostgreSQL 조합 그대로 사용 추천 (같은 스킬 재사용, 러닝커브 없음). 개인 프로젝트라 SQLite로 가볍게 가도 무방.
- **배포/실행**: 24시간 서버를 새로 띄우기보다 GitHub Actions의 scheduled workflow(cron)로 무료로 주기 실행 → 결과만 Discord Webhook으로 쏘는 구조가 비용 0원으로 가장 간단하다.
- **필터 기준 예시**: 직무 키워드(백엔드, Node.js, NestJS), 경력 조건(신입/경력무관), 채용 형태(공채/수시), 태그로 "신입공채" 별도 마킹.

## 4. 주의할 점

- 각 사이트 이용약관과 robots.txt를 먼저 확인할 것 — 상업적 재배포가 아닌 개인 알림 용도이고, 요청 간격을 충분히 두면(예: 사이트당 요청 사이 몇 초 delay) 대부분 문제되지 않지만, 공식 RSS/API가 있다면 그걸 우선 사용하는 게 안전하다.
- User-Agent를 명시하고 과도한 요청을 피할 것 (사이트당 하루 1~2회 정도의 배치 크롤링이면 충분).
- 사이트 HTML 구조는 예고 없이 바뀌므로, 파서가 깨졌을 때 알 수 있도록 에러 발생 시에도 Discord로 "OO 사이트 파싱 실패" 알림을 보내는 방어 로직을 넣으면 좋다 (이 자체가 포트폴리오에서 "운영 안정성" 케이스로 쓸 수 있다).

## 5. MVP 범위 (2~3일)

1일차: 원티드 + 점핏 파서 작성, Prisma 스키마 설계(공고 테이블: 제목/회사/링크/사이트/수집일/신입공채여부), 로컬에서 크롤링→DB 저장까지 확인.
2일차: Discord Webhook 연동, 중복 체크 로직, 키워드 필터링 완성.
3일차: GitHub Actions로 스케줄링 등록, 사람인/잡코리아 파서 추가, 실제로 며칠 돌려보며 안정화.

## 6. Claude Code에 전달할 프롬프트 (그대로 붙여넣기)

```
Node.js + TypeScript로 "DevScout"이라는 이름의 채용 공고 알림 봇을 만들고 싶어.
레포 이름은 devscout으로 할 거야.

- 원티드, 점핏 두 사이트부터 시작 (이후 사람인, 잡코리아, 랠릿, 로켓펀치, 캐치, 자소설닷컴 추가 예정)
- Playwright로 각 사이트에서 "백엔드/Node.js/신입" 관련 공고 목록을 수집
- Prisma + PostgreSQL(또는 SQLite)에 공고(제목, 회사, 링크, 사이트, 수집일시, 신입공채 여부)를 저장하고 이미 저장된 공고는 중복 알림하지 않도록 처리
- 새로 발견된 공고만 Discord Webhook으로 전송 (제목/회사/링크/신입공채 여부 태그 포함)
- 사이트별 파서는 site-adapter 패턴으로 분리해서 나중에 사이트를 쉽게 추가할 수 있게 설계
- 크롤링 실패 시에도 "OO 사이트 파싱 실패"를 Discord로 알림
- 나중에 GitHub Actions로 스케줄 실행할 예정이니 CLI 진입점(예: npm run crawl)으로 한 번 실행하면 전체 파이프라인이 끝나는 구조로 작성해줘

먼저 프로젝트 구조와 Prisma 스키마 설계안부터 제안해줘.
```

## 7. 설계 결정 사항 (Claude Code와의 논의 반영, 2026-07-14)

### 7.1 스택 확정
- 패키지 매니저: **pnpm**
- DB: **PostgreSQL** (Docker Compose로 로컬 실행, 배포 시 동일 프로토콜 유지)
- ORM: Prisma
- 프론트엔드 없음 — `pnpm crawl` 한 번 실행으로 끝나는 CLI 스크립트. GitHub Actions가 스케줄에 맞춰 이 스크립트를 실행하는 구조이며, 상시 서버 프로세스는 없음.

### 7.2 아키텍처 요약
```
CLI 진입점 (cli.ts, `pnpm crawl`)
  → pipeline.ts 오케스트레이션
    → adapters/*.adapter.ts (site-adapter 패턴. 사이트가 공개 JSON API를 제공하면 fetch로 직접 호출, 아니면 Playwright로 렌더링 후 파싱 — 어댑터 내부 구현 디테일이라 사이트마다 다를 수 있음. 원티드는 비공식 JSON API(`/api/v4/jobs`) 직접 호출로 확정, 2026-07-14)
    → crawler/run-adapter.ts (사이트별 try/catch, 실패 시 CrawlFailure 기록 + Discord 실패 알림)
    → filter/keyword-filter.ts (백엔드/Node.js/신입 키워드 매칭 + 지역/경력 조건 필터링)
    → db/job-repository.ts (url @unique 기준 신규 여부 판단 — 별도 "신규" 컬럼 없이 DB 존재 여부로 판단)
    → notify/discord.ts (신규 공고만 알림)
```

### 7.3 Prisma 스키마 (최종)
```prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}

enum Site {
  WANTED
  JUMPIT
  SARAMIN
  JOBKOREA
  RALLIT
  ROCKETPUNCH
  CATCH
  JASOSEOL
}

model JobPosting {
  id              String    @id @default(cuid())
  site            Site
  title           String
  company         String
  url             String    @unique
  isNewGradHiring Boolean   @default(false) // 대기업 정기 "신입 공채" 여부 (개별 신입 포지션과는 별개 개념)
  postedAt        DateTime?            // 공고가 사이트에 실제 등록된 시각. 사이트가 제공할 때만 채움 (nullable — 원티드 등 다수 사이트가 API/HTML에 등록 시각을 노출하지 않음, 2026-07-14 확인)
  deadlineAt      DateTime?            // 공고 마감일. 사이트가 제공할 때만 채움 (nullable — 원티드는 상시채용 위주라 대부분 null, 점핏은 대부분 값 있음)
  collectedAt     DateTime  @default(now())  // DevScout이 수집(스크랩)한 시각

  @@index([site])
}

model CrawlFailure {
  id         String   @id @default(cuid())
  site       Site
  message    String
  occurredAt DateTime @default(now())
}
```

**필드 설계 노트**
- "새로 발견된 공고인지"는 컬럼이 아니라 로직으로 판단: 크롤링된 url이 DB에 없으면 신규 → 저장 + 알림, 있으면 스킵.
- `isNewGradHiring`은 "새로 발견됨"이 아니라 "신입 공채(대기업 정기 채용 이벤트)"를 뜻하는 태그. 혼동 방지를 위해 이 이름으로 확정.
- `postedAt`과 `collectedAt`은 다른 시각: 전자는 공고 등록 시각, 후자는 DevScout이 스크랩한 시각. `postedAt`은 추후 "실제 등록 시각 데이터가 쌓이면 스케줄을 재튜닝"하는 데 사용할 계획이었으나, **원티드 공식/비공식 API를 확인해본 결과 등록 시각을 제공하지 않아 nullable로 확정** (2026-07-14). 사이트가 값을 제공하면 채우고, 아니면 null로 둔 채 `collectedAt`만 신뢰.
- **지역/경력 조건은 DB에 저장하지 않음.** 원하는 지역·경력(신입) 조건은 `filter/keyword-filter.ts`에서 크롤링 직후 필터링 기준으로만 사용하고, 조건에 안 맞는 공고는 애초에 DB에 저장되지 않음.

### 7.4 GitHub Actions 스케줄 (3일차 작업, KST → UTC 변환)
| 목적 | KST | UTC cron |
|---|---|---|
| 자정 대기업 공채 캐치 | 00:10 | `10 15 * * *` |
| 출근 직후 일반 공고 | 09:00 | `0 0 * * *` |
| 퇴근 전 마지막 배치 | 18:00 | `0 9 * * *` |

### 7.5 향후 계획
- `postedAt` 데이터가 누적되면, 사이트별로 공고가 몰리는 시간대를 분석해 위 cron 스케줄을 재조정하는 것을 아주 나중 과제로 남겨둠.
- 원하는 직무·지역·경력 조건은 `filter/keyword-filter.ts`의 매칭으로 처리(설정 파일로 분리 예정), 필요 시 사이트 자체의 검색/카테고리 쿼리 파라미터도 활용 가능.
- 알림 on/off 기능: 취업 등으로 더 이상 알림이 필요 없어졌을 때 끌 수 있는 스위치. 구현 방식(env 변수, DB 설정, CLI 플래그 등) 미정, 나중에 구체화.

### 7.6 원티드 어댑터 구현 노트 (2026-07-14)
- 원티드 목록 API(`/api/v4/jobs`)를 실제 조사해서 확정한 쿼리 파라미터: `years=0`(신입), `locations=seoul.all`(서울 전체). 직무 카테고리(백엔드/Node.js 등)는 API 파라미터로 안정적으로 안 걸려서, 카테고리 필터링은 `filter/keyword-filter.ts`에서 공고 제목 텍스트 매칭으로 처리.
- 상세 API(`/api/chaos/jobs/v4/{id}/details`) 확인 결과 `annual_from`/`annual_to`는 급여가 아니라 **경력 연차 범위**임 (예: `annual_from: 2` = 경력 2년 이상). `company.position_count`는 경력이 아니라 그 회사가 채용 중인 포지션 개수.
- 검색 조건(`years`, `locations` 등)은 **`.env` 환경변수가 아니라 어댑터 코드 안 상수로 관리**하기로 확정. 이유: 이 값들은 비밀정보가 아니라 검색 조건이고, 여러 필드가 조합된 구조라 env보다 코드가 자연스러우며, 코드로 두면 git 커밋 이력으로 "왜 조건을 바꿨는지" 추적 가능. `.env`는 `DATABASE_URL`, `DISCORD_WEBHOOK_URL`처럼 진짜 비밀값 전용으로 역할을 유지.
- `postedAt`은 원티드 API가 등록 시각을 제공하지 않아 항상 `null`로 채움 (7.3의 nullable 결정과 일치).
- `isNewGradHiring`은 제목에 "공채" 포함 여부로 최소 판단하는 휴리스틱만 적용. 원티드는 상시채용 중심 플랫폼이라 대부분 false로 나오는 게 정상이며, 이 필드가 실제로 의미 있게 쓰이는 건 나중에 캐치/자소설 어댑터를 추가할 때.
- 검색 조건은 `src/config/search-config.ts`로 분리 (2026-07-14). 사이트마다 API 파라미터 포맷이 달라 공통 키로 억지로 묶지 않고, 사이트별로 원본 값을 그대로 담는 구조로 확정 — 나중에 주변 사람에게 다른 조건(지역/경력/직무)으로 넘길 때 이 파일만 바꾸면 됨. 정렬 파라미터처럼 사람마다 바뀌지 않는 고정값은 config로 빼지 않고 어댑터 내부 리터럴로 유지.

### 7.7 마감일(`deadlineAt`) 필드 추가 결정 (다른 스레드 논의, 2026-07-14)
- `JobPosting`에 `deadlineAt DateTime?` 추가. 근거: (1) 알림 자체에 "D-3" 같은 마감 임박 표시를 넣으면 클릭 안 해도 우선순위 판단 가능, 특히 신입 공채는 마감이 타이트한 경우가 많아 실사용 가치가 큼. (2) 나중에 "마감 임박순 정렬", "이번 주 마감 필터", "마감 하루 전 리마인드" 같은 기능으로 확장 가능한데 지금 안 쌓아두면 소급 적용이 안 됨. (3) `postedAt`처럼 nullable 패턴이 이미 있어서 스키마 부담이 적음.
- 사이트별로 제공 여부가 다름: 원티드는 상시채용 위주라 `due_time`이 대부분 null, 점핏은 `closedAt`이 대부분 채워짐(2026-07-14 실제 API 확인). 값이 없으면 null로 두고 강제하지 않음.

### 7.8 점핏 어댑터 구현 노트 (2026-07-14)
- **점핏(jumpit.co.kr)이 사람인(saramin)에 흡수되어 `jumpit.saramin.co.kr`로 리다이렉트됨.** 기획서 2장의 "원티드/점핏/사람인" 대상 사이트 목록상 점핏과 사람인이 이제 같은 회사 소속이라는 뜻 — 사람인 어댑터를 나중에 추가할 때 API 구조가 점핏과 유사하거나 공유될 가능성 있음, 참고할 것.
- 목록 API: `https://jumpit-api.saramin.co.kr/api/positions`. 확정 쿼리 파라미터: `jobCategory=1`(서버/백엔드 개발자), `career=0`(신입), `locationTag=101000`(서울 전체), `sort=reg_dt`.
- 상세페이지 URL 패턴: `https://jumpit.saramin.co.kr/position/{id}` (실제 접속 확인됨).
- 응답에 `minCareer`/`maxCareer`(경력 연차), `newcomer`(신입 여부로 추정되는 필드) 존재하지만, 정확도 검증 전이라 `isNewGradHiring` 판단에는 아직 미사용 — 원티드와 동일하게 제목의 "공채" 키워드 매칭만 사용.
- 등록/게시 시각(`postedAt`에 해당하는 필드) 없음 — 원티드와 동일하게 항상 null.

## 8. 다중 사용자 확장 로드맵 (아주 나중에)

지금은 "나 혼자 쓰는 버전"에만 집중하고, 아래는 나중에 확장할 때를 위한 방향 메모.

### 8.1 단계별 진행 순서
1. **1단계 (현재)**: 나 혼자 쓰는 버전으로 안정화. 필터 조건은 `filter/keyword-filter.ts`에 하드코딩.
2. **2단계**: 주변 개발자 3~5명에게 원하는 필터 조건(직무/지역/경력)을 직접 물어보고, 그 내용을 그대로 코드에 반영해서 넘겨줌. 처음부터 범용 설정 기능을 만들지 않고 소수 실사용자 니즈를 먼저 검증하는 방식 — 초기 스타트업들이 흔히 쓰는 접근이고, "혼자 쓰려던 도구를 주변 피드백 받아 확장했다"는 포트폴리오 스토리로도 활용 가능.
3. **3단계**: 패턴이 보이면 일반화 — 사용자별로 필터 조건과 Discord 웹훅 URL을 직접 관리할 수 있게 구조 확장.

### 8.2 3단계에서 필요한 구조 변경 (지금 스키마에는 반영하지 않음)
- Prisma에 `User` 모델 추가 — 사용자별 필터 조건(키워드, 신입/경력, 직무)과 자신의 Discord 웹훅 URL 저장.
- 수집된 공고를 사용자별 필터에 매칭시켜, 매칭되는 유저의 웹훅으로만 개별 전송.
- 필터 입력/수정 인터페이스 옵션:
  - **Discord 슬래시 커맨드** (예: `/필터설정 키워드:백엔드,Node.js`) — 별도 웹 프론트 없이 이미 쓰는 Discord 안에서 해결. Discord Bot API 다루는 경험이 추가되어 기술적으로도 볼거리가 됨.
  - 필요하면 나중에 Next.js로 간단한 설정 페이지를 만들어 프론트 포트폴리오까지 확장 가능.

**결론**: 지금 당장은 1단계(나 혼자 쓰는 버전)만 잘 끝내는 데 집중. User 모델/멀티테넌시 구조는 지금 스키마·아키텍처에 미리 넣지 않는다.
