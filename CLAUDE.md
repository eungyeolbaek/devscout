# DevScout — Claude Code 작업 규칙

## 문서 구조
- `docs/기획.md`: 프로젝트 기획서. 목표, 대상 사이트, 아키텍처, Prisma 스키마 등 **설계 결정**을 담는다. 결정된 내용은 함부로 뒤집지 말고, 바뀌면 해당 섹션을 갱신한다.
- `docs/PROGRESS.md`: **진행 상황 로그** (gitignore 처리, 로컬 전용). 무엇을 완료했고, 다음에 뭘 해야 하는지를 담는다. 기획서와 분리해서 관리한다.

## 갱신 규칙
- 설계 결정(스택, 스키마, 아키텍처, 스케줄 등)이 새로 확정되거나 바뀌면 → `docs/기획.md`를 갱신한다.
- 작업을 완료하거나 다음 할 일이 바뀌면 → `docs/PROGRESS.md`를 갱신한다.
- 세션/스레드가 새로 시작되면 위 두 파일을 먼저 읽고, 이미 답변/결정된 내용을 사용자에게 다시 묻지 않는다.
- 사용자가 명시적으로 "기록해줘", "md에 반영해줘"라고 하지 않아도, 설계 결정이나 진행 상황에 변화가 생기면 알아서 해당 파일을 갱신한다.

## 커밋 규칙
- 커밋은 사용자가 직접 실행한다. Claude Code는 `git commit`을 대신 실행하지 말고, 커밋 메시지만 제안한다.
- 커밋 메시지는 한글로, 제목(머릿말) 한 줄만 준다. 본문(body)이나 Co-Authored-By 같은 트레일러는 붙이지 않는다.
- 제목 앞에 conventional commit 타입 접두어를 반드시 붙인다 (예: `chore:`, `feat:`, `fix:`, `docs:`, `refactor:`). 접두어 없이 메시지만 주지 않는다.
- 커밋 메시지는 항상 코드 블록(```)으로 감싸서 준다 — 복사 버튼이 바로 뜨도록. 본문에 텍스트로 풀어쓰지 않는다.
- 하나의 작업 단위가 끝나서 커밋하기 좋은 지점이 되면, 사용자가 묻지 않아도 먼저 알려주고 커밋 메시지를 제안한다. "지금 커밋하는 게 좋겠다"는 판단이 서면 바로 알린다.

## 프로젝트 확정 스택 (요약, 상세는 기획서 참고)

- GitHub Actions로 스케줄 실행 예정 (KST 00:10 / 09:00 / 18:00)
